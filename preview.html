Technical Design: Office LLM Harness via VSTO, MCP and Open WebUI

Design proposal for a Windows desktop Office add-in harness that exposes controlled document interaction tools to Open WebUI through a local MCP server, enabling LLM-assisted workflows in Word, Excel, PowerPoint and Outlook.

1. Executive summary

Recommended Pattern B Local MCP server Windows only

The recommended implementation is a VSTO-based Office harness that starts or connects to a local MCP server. Open WebUI connects to this MCP server as an external tool provider. The MCP server exposes controlled, document-aware tools such as reading the current selection, extracting slide outlines, reading worksheet ranges, drafting Outlook replies and applying approved document edits.

This keeps Office automation local to the user session, avoids exposing raw COM automation to the model and allows Open WebUI to use the tools through its native MCP integration. For an MVP, the MCP endpoint can bind to 127.0.0.1 and use no authentication. For production, add a per-session token and explicit confirmation for destructive actions.

2. Goals and non-goals

Goals

  • Provide LLM capabilities inside Word, Excel, PowerPoint and Outlook on Windows desktop Office.
  • Expose document interaction through capability-based tools, not arbitrary code execution.
  • Use Open WebUI as the LLM front end, model router and tool caller.
  • Support local document context extraction and safe document mutation.
  • Preserve user control through previews, diffs and confirmation gates.
  • Make the architecture extensible enough to add more Office hosts later.

Non-goals

  • Support Office for Web, Office for Mac or mobile Office in the first implementation.
  • Expose unrestricted Office COM automation to the model.
  • Send entire documents by default.
  • Let the model send emails, delete content or overwrite large ranges without explicit user confirmation.
  • Implement a general-purpose enterprise DLP system in the MVP.

3. Assumptions

Area Assumption Impact
Office runtime Users run Windows desktop Office with VSTO support. Allows deep access to Word, Excel, PowerPoint and Outlook object models.
Open WebUI Open WebUI version supports native MCP Streamable HTTP external tools. Local MCP server can be registered directly instead of being bridged via OpenAPI.
Networking MCP server runs on the same Windows host as Office, bound to localhost. Remote network exposure is avoided for the MVP.
Open WebUI hosting Open WebUI may run locally, in Docker or on a remote server. Docker requires host.docker.internal; remote hosting cannot directly reach localhost without a tunnel or agent.
Authentication MVP can skip MCP auth if localhost-only; production should add a session token. Simpler MVP, but local process abuse remains a risk without tokenization.

Recommended process model

Use one shared helper process for the MCP server instead of embedding the MCP server inside every VSTO add-in. Each Office host add-in communicates with the helper over local IPC.

VSTO add-in

Lives inside Office. Owns UI, Office object-model access, selection state and confirmation workflows.

MCP helper process

Lives outside Office. Owns MCP transport, tool registry, request validation, audit logging and coordination.

Why not expose raw COM?

Direct model access to COM would create uncontrolled mutation risk, unstable execution and a difficult security boundary. The MCP server must expose narrow, intentional tools with typed inputs and bounded outputs.

5. Components

Component Responsibilities Technology suggestion
Office host add-ins Ribbon/task pane UI, Office COM access, selected document/session state, confirmation dialogs. C# VSTO Add-ins targeting .NET Framework.
Office adapter library Normalizes document operations across Office hosts and implements app-specific adapters. Shared C# library referenced by VSTO projects.
Local MCP server Exposes MCP tools over Streamable HTTP and routes tool calls to the active Office adapter. .NET 8 worker process or minimal HTTP service.
IPC channel Connects MCP helper to the in-process VSTO add-ins. Named pipes preferred; localhost HTTP acceptable for MVP.
Open WebUI LLM chat interface, model routing, MCP external tool registration. Existing Open WebUI deployment.
Audit store Records tool calls, inputs, summaries, user confirmations and resulting changes. Local JSONL or SQLite for MVP; central logging later.
Implementation preference: keep model interaction inside Open WebUI and Office interaction inside the local harness. The MCP server is the boundary between the two.

6. MCP interface

Endpoint

http://127.0.0.1:<port>/mcp

If Open WebUI runs in Docker on the same workstation, register the endpoint as: 

http://host.docker.internal:<port>/mcp

Open WebUI external tool registration

Setting MVP value Production value
Type MCP / Streamable HTTP MCP / Streamable HTTP
URL http://127.0.0.1:<port>/mcp Same, or managed local agent endpoint.
Authentication None Bearer token or custom header with per-session secret.
Scope Local developer machine. Per user, centrally managed deployment policy.

Tool naming convention

Use stable, explicit names. Prefix generic tools with office_ and app-specific tools with word_, excel_, powerpoint_ or outlook_.

Standard tool response envelope

{
  "ok": true,
  "app": "PowerPoint",
  "documentId": "local-session:powerpoint:active",
  "result": {},
  "warnings": [],
  "requiresConfirmation": false,
  "auditId": "20260514-153500-00042"
}

Error response envelope

{
  "ok": false,
  "errorCode": "NO_ACTIVE_DOCUMENT",
  "message": "No active PowerPoint presentation is available.",
  "recoverable": true,
  "auditId": "20260514-153500-00043"
}

Confirmation contract

Tools that can materially change a document should return a preview instead of directly applying the change unless the request contains a valid confirmation token. 

{
  "ok": true,
  "requiresConfirmation": true,
  "confirmation": {
    "confirmationId": "confirm-9f2a",
    "summary": "Replace text in 3 shapes on slide 4.",
    "diffPreview": [
      {
        "target": "slide:4/shape:Title 1",
        "before": "Quarterly results",
        "after": "Q3 performance summary"
      }
    ]
  }
}

8. Security model

MVP security posture

  • MCP server binds only to 127.0.0.1.
  • No MCP authentication for local-only development.
  • No arbitrary shell execution, macro execution or raw COM dispatch exposed as tools.
  • Destructive operations return previews and require UI confirmation.
  • Tool outputs are bounded in size and avoid full-document export by default.

Production hardening

  • Generate a random per-session token at add-in startup.
  • Pass the token to Open WebUI as an MCP custom header or bearer token.
  • Rotate token whenever the Office session or MCP helper restarts.
  • Store no long-lived local secrets unless needed for managed deployment.
  • Add tool-level allowlists by Office host and user role.
  • Log every tool call with timestamp, chat/user correlation fields, document handle, mutation status and confirmation ID.
  • Block or redact sensitive data types before sending context to the model where policy requires it.

Threat model

Threat Risk Mitigation
Local malicious process calls MCP endpoint Medium if no token is used. Use per-session token in production; bind localhost only; restrict tool set.
Model overwrites document content incorrectly High for user trust. Preview/diff, confirmation gate, undo support and audit log.
Prompt injection from document content Medium to high. Treat document content as untrusted data; system prompts must instruct model not to follow embedded instructions.
Email sent unintentionally High external side effect. Disable send in MVP or require manual Outlook confirmation.
Excessive context leakage High depending on document sensitivity. Bounded context extraction, explicit export tools, redaction hooks and configurable limits.

9. Data handling

Context selection rules

  1. Prefer current selection.
  2. If no selection exists, use active slide, active worksheet region, active Word section or selected Outlook item.
  3. Never send entire workbooks, decks, documents or mailboxes by default.
  4. Apply output limits per tool, for example 32 KB default and 256 KB maximum unless explicitly approved.
  5. Return metadata separately from content to help the model choose follow-up tools.

Suggested context envelope

10. User experience

Office ribbon

  • Ask LLM: opens the task pane and sends current context.
  • Summarize: summarizes current document, slide, range or email thread.
  • Rewrite: rewrites selected text or selected objects.
  • Generate: creates a slide, section, formula, table or draft reply.
  • Review changes: shows pending model-proposed edits.

Task pane

  • Shows current Office context being shared.
  • Lets user choose whether to include selection, surrounding context, notes, formulas or comments.
  • Displays pending changes before applying them.
  • Shows audit details for the last tool calls.

Confirmation UX

The model can propose changes, but the add-in applies them only after user approval. For Word and PowerPoint, show a before/after diff. For Excel, show affected ranges, formula changes and sample changed cells. For Outlook, drafts may be created automatically, but sending requires manual approval.

11. Implementation plan

Phase 0 — Spike

  • Create a minimal local MCP Streamable HTTP server exposing office_get_active_app.
  • Create one PowerPoint VSTO add-in that reports active presentation and selected slide.
  • Connect Open WebUI to the local MCP endpoint.
  • Validate Docker networking with host.docker.internal if Open WebUI runs in Docker.

Phase 1 — PowerPoint MVP

  • Implement powerpoint_get_deck_outline, powerpoint_get_slide, powerpoint_update_shape_text and powerpoint_update_speaker_notes.
  • Add diff preview for text changes.
  • Add local audit log.
  • Add basic task pane with context preview.

Phase 2 — Word MVP

  • Implement Word outline, selection read, rewrite selection and add comment tools.
  • Use Word comments or tracked changes for review-oriented workflows.
  • Add shared document context abstraction across Word and PowerPoint.

Phase 3 — Excel MVP

  • Implement workbook map, read range, write range and write formula tools.
  • Add range-size limits and formula validation.
  • Add confirmation preview for any write operation.

Phase 4 — Outlook MVP

  • Implement current item read, thread summary and draft reply tools.
  • Keep sending disabled or gated behind manual Outlook confirmation.
  • Add optional policy filter for recipient domains and attachments.

Phase 5 — Production hardening

  • Add per-session token authentication.
  • Add installer, auto-start, crash recovery and version reporting.
  • Add centralized policy and logging hooks.
  • Add DLP/redaction extension points.
  • Add admin-configurable tool allowlist.

12. Testing strategy

Layer Tests
Tool schemas Validate MCP tool inputs, required fields, enum values and response envelopes.
Office adapters Unit-test serialization and mutation preview generation with mocked Office objects where possible.
VSTO integration Manual and automated smoke tests in Word, Excel, PowerPoint and Outlook.
Safety gates Verify destructive tools do not apply changes without confirmation tokens.
Prompt injection Use documents containing adversarial instructions and verify tool policy remains enforced.
Open WebUI integration Validate tool discovery, tool execution and error handling from chat.

13. Deployment

Developer deployment

  • Install VSTO add-ins through Visual Studio or ClickOnce for local testing.
  • Run MCP helper as a user-mode process started by the first Office add-in loaded.
  • Register MCP endpoint manually in Open WebUI External Tools.

Managed deployment

  • Package VSTO add-ins and MCP helper with a signed installer.
  • Auto-start MCP helper on Office launch only, not at system boot unless required.
  • Use fixed localhost port range or dynamic port discovery through a local manifest file.
  • Distribute Open WebUI tool configuration through admin documentation or deployment automation.
  • Sign binaries and installer packages.

Versioning

Expose a office_get_harness_status tool returning add-in versions, MCP server version, enabled tools, active Office hosts and policy version. This helps debug Open WebUI tool mismatch issues.

14. Risks and mitigations

Risk Impact Mitigation
VSTO limits platform support to Windows desktop Office. Mac/Web users excluded from MVP. Accept for MVP; design document APIs so a future Office Web Add-in can reuse server-side logic.
Open WebUI cannot reach local MCP server when hosted remotely. Pattern B breaks for remote Open WebUI. Use local Open WebUI, Docker host routing, managed local agent or reverse tunnel with auth.
Office COM automation can be fragile across app states. Tool calls may fail depending on modal dialogs or protected documents. Detect unsupported states and return recoverable MCP errors.
LLM generates poor edits. User trust and document quality risk. Use previews, comments, tracked changes and clear undo support.
Large documents exceed useful model context. Latency, cost and degraded answers. Chunking, targeted retrieval tools and bounded context extraction.

15. Decisions and open questions

Proposed decisions

  • Use Pattern B: local MCP server exposed to Open WebUI.
  • Use VSTO for deep Office desktop integration.
  • Use a helper process for MCP rather than embedding the MCP endpoint in each add-in.
  • Start MVP with PowerPoint, then Word, then Excel, then Outlook.
  • Allow unauthenticated localhost-only MCP for the spike; add per-session token before production.
  • Never expose arbitrary COM execution, shell execution or macro execution as model-callable tools.

Open questions

  • Will Open WebUI run locally on the user's machine, in Docker, or on a remote shared server?
  • Should the MCP server be one per user session or one per Office process?
  • Do we need central audit logging from day one?
  • Which identity correlation fields should Open WebUI pass to the local MCP server?
  • What is the first production-grade target workflow: slide generation, Word review, Excel formula help or Outlook drafting?
  • Which documents or email types must be blocked from model context export?
